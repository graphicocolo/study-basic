# JavaScript 公式ドキュメントの読み方

- ドキュメントを読む必然性
- ドキュメントを読むコツ

## なぜまず公式ドキュメントを読むべきか？

**一言で言うと**

一番基本的で正確な情報だから

検索や AI 経由で得られる二次情報だと、正確性や更新性に欠ける場合がある

## ドキュメントを読むコツ

- 構文セクションを重点的に読む
  - 引数の型、省略可能かどうか、返り値の型を確認する習慣をつける
- 読んだ内容を一言でメモし、ログに残す
  - `Array.from()` → 配列風オブジェクトやイテラブルから「本物の配列」を作る
  - 自分の言葉で要約すると定着しやすい
  - 後述の3つ程度のコードもログに残す
- コードを動かすサンドボックスを用意して読む
  - ドキュメントのコードを3つ程度実際に書いて動かしてみる
  - [CodeSandbox](https://codesandbox.io) VS Codeのようなエディタ環境がブラウザで動く

### さらに詳しく

- **構文セクションを重点的に読む**
  - 引数の型、省略可能かどうか、返り値の型を確認する習慣をつける
- 関連メソッドとの比較表を自分で作る
  - map, forEach, splice, など似たメソッドを比較することで違いが明確になる
- エッジケースを自分で試す
  - `[].map(x => x * 2)  // → []`
  - `[1, 2, 3].includes(undefined)  // → false`
- 関連情報セクションを活用
  - 芋づる式に知識を広げられる
- 読んだ内容を一言でメモする
  - `Array.from()` → 配列風オブジェクトやイテラブルから「本物の配列」を作る
  - 自分の言葉で要約すると定着しやすい
- コードを動かすサンドボックスを用意して読む
  - [CodeSandbox](https://codesandbox.io) VS Codeのようなエディタ環境がブラウザで動く。
  - JavaScript ならばコードエディタの Go Live ですぐにブラウザ確認可能
- 読みたい事柄が掲載されたページのコードを3つ程度実際に書いて動かしてみる
- 翻訳ツールをうまく使って読む（英語ドキュメントの場合）
- 「仕様書」リンクを怖がらない
  - MDNの各ページ下部にECMAScript仕様へのリンクがある。最初は難しく見えるが、「アルゴリズムステップ」を追うと挙動が正確に理解できる。
  - 最初は全部読まなくても、「気になる挙動があったときに該当ステップを探す」使い方で十分です
